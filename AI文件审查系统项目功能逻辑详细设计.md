# AI文件审查系统项目功能逻辑详细设计

# 一、系统整体功能逻辑架构

系统采用分层架构+流程编排模式，自下而上分为5个核心层级，各层级职责明确、协同联动，确保从文档输入到报告输出的全流程顺畅执行，具体层级划分及核心职责如下：

1. **基础设施层**：作为系统运行的基础支撑，负责全局配置管理、文件IO操作、外部接口（大模型API）通信，为上层模块提供稳定的运行环境。

2. **文档处理层**：核心数据加工层，负责将非结构化的Word文档解析、切分，转化为结构化的“切片”集合，为后续审查提供标准化的数据格式。

3. **规则与模型层**：负责审查规则的加载与映射，以及大模型的交互适配，明确审查标准、打通AI审查通道，是实现智能审查的核心支撑。

4. **审查引擎层**：逻辑编排核心，串联文档处理、规则加载、模型交互等模块，执行全流程审查逻辑，确保各环节有序衔接、高效运行。

5. **结果输出层**：负责审查结果的汇总、解析与可视化，生成可直接用于业务查看的Excel审查报告，完成最终输出。

# 二、详细功能逻辑流程

系统功能逻辑按“初始化→文档处理→规则加载→AI审查→结果输出”的顺序执行，各阶段环环相扣，形成完整的业务闭环，具体流程如下：

## 阶段1：系统初始化与环境准备

### 核心目标

加载系统全局配置，初始化运行环境，为后续所有流程提供基础支撑，避免因配置缺失、环境异常导致流程中断。

### 功能逻辑

1. **配置加载**：Spring容器启动时，通过spring-boot-configuration-processor包，将application.yml配置文件中的核心参数（API密钥、工作目录路径、大模型接口地址、提示词文件路径等），自动绑定到专用的配置属性类中，供各模块统一调用，无需重复定义。

2. **目录初始化**：调用工作目录初始化函数，检查配置文件中指定的输出目录（如“切分结果”“审查报告”文件夹）是否存在；若不存在，则自动创建对应目录，确保后续切分结果、审查报告等文件能正常保存，避免文件写入失败。

### 涉及模块

配置与基础设施模块

### 依赖包

spring-boot-starter（Spring核心包）、spring-boot-configuration-processor（配置文件绑定）

## 阶段2：Word文档解析与切分（核心数据加工）

### 核心目标

将非结构化的Word文档（含文本、表格、图片、多级列表等），转化为结构化的“切片”集合，使文档内容可被AI模型高效识别、审查，同时保存切分结果用于调试或断点续传。

### 功能逻辑

1. **文档加载**：接收外部输入的Word文件路径，调用文档加载函数，通过org.apache.poi:poi-ooxml包将Word文件（.docx格式）加载为内存中的XWPFDocument对象，便于后续解析文档内部元素。

2. **编号规则提取**：调用编号定义获取函数，从加载的文档中提取多级列表的定义规则（如“标题1对应1、2、3，标题2对应1.1、1.2，标题3对应1.1.1、1.1.2”），并将规则注入编号跟踪器类，完成编号跟踪器的初始化，为后续段落编号计算做准备。

3. **元素遍历与处理**：调用文档块遍历函数，按文档顺序扫描所有元素，区分段落和表格，分别进行针对性处理：
        

    - 若为段落：调用段落解析函数，识别段落样式（判断是标题还是正文）、通过编号跟踪器实时计算当前段落的完整编号（如1.2.3）、提取段落纯文本内容；若段落中包含图片，提取图片数据并保存到本地指定目录，同时记录图片路径，便于后续AI审查时关联查看。

    - 若为表格：调用表格转HTML/文本函数，将Word表格的行列结构、单元格内容、合并单元格等信息，转换为HTML字符串（保留表格原有格式），避免表格结构丢失，确保AI模型能准确识别表格内容。

4. **文档切分**：执行文档切分主函数，维护一个“当前切片”对象，按预设的标题级别（如标题1、标题2）执行切分逻辑：
        

    - 当遇到指定级别的标题时，若“当前切片”已存在（非首次切分），将其存入切片集合；随后创建新的切片对象，将当前标题设置为切片标题。

    - 将后续解析完成的段落文本、HTML表格，依次追加到“当前切片”的内容中，直至遇到下一个同级标题，完成一个切片的构建。

5. **切分结果暂存**：调用切片保存函数，通过com.fasterxml.jackson.core:jackson-databind包，将内存中所有切片组成的Map集合（key为切片标识，如section_1、section_2；value为切片对象，含标题、内容），以格式化JSON的形式写入硬盘，保存路径为“工作目录/2_切分结果.json”，用于后续调试、断点续传（若流程中断，可直接加载JSON恢复切片数据，无需重新解析文档）。

### 涉及模块

文档解析与切分模块、切分结果持久化模块

### 依赖包

org.apache.poi:poi-ooxml（Word文档操作）、com.fasterxml.jackson.core:jackson-databind（JSON处理）

## 阶段3：审查规则与映射关系建立

### 核心目标

明确“文档的哪一部分（切片）需要审查什么内容”，建立章节与检查项的对应关系，为AI审查提供明确的审查标准和范围。

### 功能逻辑

1. **配置文件读取**：调用审查配置加载函数，读取指定路径下的规则配置文件（如config_noelecmag.json），该文件包含章节特征、检查项列表、检查标准等核心信息。

2. **映射关系构建**：通过com.fasterxml.jackson.core:jackson-databind包，将JSON格式的配置文件反序列化为Java配置对象；随后执行章节-检查项映射函数，在内存中构建一个Map<章节特征, 检查项列表>的映射表（例如：“范围”章节 → 检查“是否包含完整性说明”“是否明确适用场景”；“技术参数”章节 → 检查“参数取值合规”“单位标注准确”），确保每个切片都能匹配到对应的审查项。

### 涉及模块

配置映射与规则加载模块

### 依赖包

com.fasterxml.jackson.core:jackson-databind（JSON反序列化）

## 阶段4：AI驱动的内容审查（核心业务逻辑）

### 核心目标

调用大模型，按“切片+检查项”的维度，逐一对文档内容进行智能审查，提取审查结论、问题描述和修改建议，形成结构化的审查结果。

### 功能逻辑

1. **审查引擎初始化**：创建审查引擎实例，注入大模型客户端（如SiliconFlowClient、Qwen3Client）和提示词模板，确保审查引擎能正常调用大模型、生成符合要求的提示词。

2. **双层循环遍历**：采用双层循环，确保每个切片的每个检查项都能被审查，不遗漏任何内容：
        

    - 外层循环：遍历阶段2生成的所有文档切片，依次处理每个切片。

    - 内层循环：根据当前切片的标题/特征，通过阶段3构建的映射表，查询出该切片对应的所有检查项，依次处理每个检查项。

3. **单条审查执行**：针对每个切片+每个检查项，执行完整的AI审查流程，步骤如下：
        

    - 构建提示词：调用提示词构建函数，将系统提示词（设定AI审查角色，如“专业文档审查员，严格按检查标准审查内容”）、检查项要求（明确审查标准，如“检查内容是否完整，无遗漏关键信息”）、当前切片内容（文本+HTML表格）拼接为完整的Prompt，确保AI能清晰理解审查需求。

    - 调用模型：根据当前切片内容类型（纯文本/复杂表格），通过模型路由函数（可选）选择对应的大模型客户端实现类，通过spring-boot-starter-webflux包提供的WebClient（或spring-web包的RestTemplate），发送HTTP POST请求到大模型API（携带API密钥、Prompt等参数），获取模型返回的JSON格式审查结果。

    - 解析结果：调用审查结果解析函数，从模型返回的JSON中，提取出“审查结论（通过/不通过）”“问题描述（如‘缺少关键参数单位’）”“修改建议（如‘补充参数对应的单位，如MPa、mm’）”，封装为结构化的ReviewResult对象，便于后续汇总和报告生成。

4. **结果收集**：将所有封装后的ReviewResult对象，存入一个全局的review_collector集合中，按“切片标识+检查项ID”进行区分，确保结果不重复、可追溯。

### 涉及模块

大模型交互客户端模块、审查引擎核心模块

### 依赖包

spring-boot-starter-webflux（WebClient）或spring-web（RestTemplate）、com.fasterxml.jackson.core:jackson-databind（JSON解析）

## 阶段5：审查结果汇总与Excel报告生成

### 核心目标

将AI审查的结构化结果，转化为直观、可读的Excel业务报告，便于用户查看审查情况、跟踪问题整改。

### 功能逻辑

1. **Excel初始化**：调用Excel总结生成函数，通过org.apache.poi:poi-ooxml包创建XSSFWorkbook工作簿，并在工作簿中创建“审查总结”工作表，作为报告的核心载体。

2. **表头创建**：在工作表的第一行，创建固定表头，明确报告列名，便于用户快速理解内容，表头内容如下：章节名称、检查项、审查结果、修改建议。

3. **数据填充**：遍历review_collector集合，从工作表第二行开始，将每个ReviewResult对象的对应字段（章节名称、检查项名称、审查结论、修改建议）逐列写入单元格，确保数据准确对应、无遗漏。

4. **格式优化与保存**：自动调整工作表的列宽，适配内容长度（避免文本截断）；将工作簿写入指定路径（如“工作目录/审查报告.xlsx”），完成报告生成。

### 涉及模块

Excel报告生成模块

### 依赖包

org.apache.poi:poi-ooxml（Excel操作）

## 阶段6：全流程编排（总控逻辑）

### 核心目标

通过一个统一入口，串联上述所有阶段，实现流程的自动化执行，同时处理异常情况，确保系统稳定运行。

### 功能逻辑

1. 接收外部触发：接收API请求、定时任务或手动操作触发的审查任务，获取输入Word文档的路径（核心输入参数）。

2. 流程串联执行：按顺序调用各阶段的核心函数，形成完整的业务链路：阶段1（初始化）→ 阶段2（文档切分）→ 阶段3（规则加载）→ 阶段4（AI审查）→ 阶段5（Excel输出），确保各环节有序衔接、数据正常流转。

3. 异常处理：捕获上述任一阶段的异常（如文档解析失败、模型调用超时、文件写入失败等），记录详细日志（异常信息、发生阶段、当前处理的内容），终止流程，并向外部返回明确的错误信息（如“文档解析失败，请检查文件格式是否为.docx”），便于问题排查。

4. 流程终止与反馈：若流程全部执行成功，返回Excel审查报告的文件路径，供用户下载、查看；若流程异常终止，返回错误信息，提示用户处理异常后重新触发任务。

### 涉及模块

流程编排模块（Facade模式）

### 依赖包

spring-context（@Service注解，依赖注入）

# 三、关键数据流转逻辑

系统数据从输入到输出，经历多个中间态转换，确保数据可追溯、可复用，具体流转路径如下：

1. **输入**：用户提供的Word文档（.docx格式）路径，是系统的初始数据来源。

2. **中间态1**：Map<String, DocumentSection>（切片集合），由阶段2生成，每个元素包含切片标识、标题、结构化内容（文本+HTML表格），是AI审查的核心数据。

3. **中间态2**：JSON文件（2_切分结果.json），由阶段2生成，是切片集合的持久化备份，用于调试、断点续传。

4. **中间态3**：Map<String, List<CheckItem>>（章节与检查项映射表），由阶段3生成，明确审查范围和标准，指导AI审查执行。

5. **中间态4**：List<ReviewResult>（结构化审查结果集合），由阶段4生成，包含每个切片、每个检查项的完整审查信息，是报告生成的核心数据。

6. **输出**：Excel审查报告（.xlsx格式），由阶段5生成，是系统的最终输出，供用户查看、使用。

# 四、核心注意事项

1. 文档格式限制：系统仅支持.docx格式的Word文档，若输入其他格式（如.doc、.pdf），需提前转换，否则会导致文档解析失败。

2. 配置参数有效性：API密钥、大模型接口地址、工作目录路径等配置参数需准确填写，否则会导致模型调用失败、文件无法保存等问题。

3. 异常处理：各阶段需做好异常捕获，尤其是模型调用（可能存在超时、接口报错）、文件IO（可能存在权限不足）等场景，避免流程崩溃。

4. 数据一致性：切片集合、审查结果集合需保证数据一致性，避免出现切片缺失、审查结果重复或遗漏的情况。

5. 性能优化：对于大型Word文档（如超过100页），可考虑分批次解析、切分，避免内存溢出；模型调用可引入缓存机制，重复的切片+检查项可复用审查结果，提升效率。
