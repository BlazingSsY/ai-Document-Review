# AI 文件审查系统产品需求文档

**版本**：V1.0
**文档状态**：已发布

---

## 1. 产品概述

### 1.1 产品定位

AI 文件审查系统是一款面向企业和机构的智能文档审查平台，通过接入多种大语言模型（LLM），对 Word 格式文档进行自动化审查分析，依据预设的审查规则发现文档中的问题并提出改进建议，帮助用户提高文档审查效率和质量。

### 1.2 目标用户

| 用户角色 | 使用场景 |
|---------|---------|
| 项目主管 | 管理审查团队、配置系统、分配权限 |
| 管理员 | 配置审查规则与场景、管理 AI 模型 |
| 普通用户 | 上传文档、执行审查、查看审查结果 |

### 1.3 产品目标

- 将传统人工文档审查的效率提升 5-10 倍
- 通过标准化规则确保审查质量的一致性
- 支持多种 AI 模型，降低对单一厂商的依赖
- 提供实时审查进度反馈，提升用户体验

## 2. 功能需求

### 2.1 用户认证与授权

#### 2.1.1 用户注册

- 用户通过邮箱和密码注册账号
- 密码需要确认输入，两次输入一致方可注册
- 注册成功后自动登录并跳转到工作台
- 新注册用户默认角色为"普通用户"

#### 2.1.2 用户登录

- 通过邮箱和密码登录
- 登录成功后获取 JWT Token（Access Token + Refresh Token）
- Access Token 有效期 1 小时，Refresh Token 有效期 7 天
- Token 过期后自动刷新，无需重新登录

#### 2.1.3 个人中心

- 查看个人基本信息（姓名、邮箱、角色）
- 修改登录密码（需输入原密码验证）

#### 2.1.4 权限体系

系统设置三级角色权限：

| 权限项 | 普通用户 | 管理员 | 项目主管 |
|--------|---------|--------|---------|
| 上传文档并发起审查 | Y | Y | Y |
| 查看、取消、删除自己的审查任务 | Y | Y | Y |
| 查看审查结果详情 | Y | Y | Y |
| 上传审查规则文件 | - | Y | Y |
| 创建/编辑/删除规则库 | - | Y | Y |
| 创建/编辑/删除审查场景 | - | Y | Y |
| 配置 AI 模型 | - | Y | Y |
| 管理用户（增删改、角色分配） | - | - | Y |
| 为用户分配规则库访问权限 | - | - | Y |

### 2.2 工作台（Dashboard）

工作台是用户登录后的主页面，集中展示审查工作的核心信息。

#### 2.2.1 统计概览

页面顶部展示四个统计卡片：

| 卡片 | 内容 | 说明 |
|------|------|------|
| 审查总数 | 当前用户提交的所有审查任务总数 | 蓝色主题 |
| 已完成 | 状态为"已完成"的任务数 | 绿色主题 |
| 进行中 | 状态为"处理中"的任务数 | 紫色主题，有任务处理中时图标旋转 |
| 今日审查 | 今日提交的审查任务数 | 橙色主题 |

#### 2.2.2 审查任务列表

- 分页展示当前用户的所有审查任务（每页 10 条）
- 支持按状态筛选（待处理、处理中、已完成、失败、已取消）
- 每条任务显示：文件名、AI模型、状态标签、发现问题数、创建时间
- 操作按钮：
  - **查看详情**：跳转到审查详情页（待处理状态不可点击）
  - **重新审查**：对已完成/失败/已取消的任务创建一个新的审查任务，复用原任务的文件、场景和模型
  - **取消**：取消待处理或处理中的任务
  - **删除**：删除非处理中的任务（需确认）

#### 2.2.3 新建审查

点击"新建审查"按钮弹出模态窗口，分两步完成：

**第一步：上传文件**
- 支持拖拽上传或点击选择
- 仅支持 Word 文档格式（.doc、.docx）
- 最大文件大小 200MB
- 显示已选文件名和文件大小

**第二步：配置参数**
- 选择审查场景（下拉列表，支持搜索）
- 选择 AI 模型（下拉列表，仅显示已启用的模型）
- 点击"提交审查"后自动关闭窗口，任务列表立即刷新

### 2.3 审查详情页（Review Workspace）

点击任务列表中的"查看详情"进入审查详情页面。

#### 2.3.1 任务信息

- 文件名、使用的 AI 模型、创建时间
- 任务状态标签（颜色编码）
- 审查进度条（0-100%）

#### 2.3.2 实时日志

- 独立的日志面板，实时显示审查过程中的每一步操作
- 日志内容包括：文档解析进度、规则加载情况、文档切片信息、每个章节的审查进度、错误信息
- 通过 WebSocket 实时更新，无需刷新页面
- 日志条目带时间戳

#### 2.3.3 审查结果

审查完成后展示：
- 总评分（如果 AI 返回了评分）
- 发现问题总数
- 按章节分组的问题列表
- 每个问题包含：严重程度（高/中/低）、问题类别、原始文本引用、改进建议

#### 2.3.4 操作按钮

- **重新审查**：使用相同的文件、场景、模型重新发起审查
- **取消审查**：对处理中的任务发送取消指令
- **返回工作台**：回到任务列表

### 2.4 规则与场景管理

规则与场景合并在同一个页面中，通过 Tab 页签切换。

#### 2.4.1 审查场景（第一个 Tab）

审查场景是一组规则库的组合，定义了文档审查的标准和范围。

**场景列表**
- 分页表格展示所有场景
- 显示：场景名称、描述、创建者、关联规则库数量
- 操作：查看详情、编辑、删除（管理员/主管权限）

**创建场景**
- 输入场景名称（必填）
- 输入场景描述（选填）
- 选择关联的规则库（复选框列表，至少选择一个）
- 场景关联的是规则库，不是单条规则

**编辑场景**
- 修改场景名称和描述
- 增加或移除关联的规则库

**查看场景详情**
- 展示场景基本信息
- 列出关联的所有规则库

#### 2.4.2 审查规则（第二个 Tab）

**规则列表**
- 分页表格展示所有规则
- 显示：规则名称、文件类型（md/json）、创建者、所属规则库
- 操作：删除（管理员/主管权限）

**上传规则**
- 支持 Markdown（.md）和 JSON（.json）两种格式
- 上传时可选择归属的规则库
- 系统自动检测文件类型并验证格式有效性

**规则库管理**
- 创建规则库：输入名称和描述
- 编辑规则库：修改名称和描述
- 删除规则库：级联删除库中所有规则

### 2.5 模型配置管理

管理接入的 AI 大语言模型。

#### 2.5.1 模型列表

- 分页表格展示所有已配置的模型
- 显示：模型名称、厂商、启用状态
- 操作：编辑、启用/禁用切换、删除

#### 2.5.2 添加/编辑模型

配置项：

| 字段 | 必填 | 说明 |
|------|------|------|
| 模型名称 | Y | 显示名称（如"GPT-4o"） |
| 厂商 | Y | 预设：OpenAI、Anthropic、Moonshot、百度、阿里、讯飞；支持自定义输入 |
| Model Key | - | 实际调用的模型标识（如"gpt-4o"），不填则使用模型名称 |
| API Endpoint | Y | API 接口地址，系统根据厂商自动拼接完整路径 |
| API Key | Y | 调用密钥（界面显示脱敏，编辑时不重新输入则保持原值） |
| Max Tokens | - | 最大生成 Token 数，默认 4096 |
| Temperature | - | 生成温度参数（0-1），控制输出的随机性。值越低越确定，值越高越多样 |
| 启用状态 | - | 是否在审查时可选，默认启用 |

#### 2.5.3 厂商 API 路径自动拼接规则

系统根据选择的厂商自动拼接完整的 API 请求地址：

| 厂商 | 用户输入示例 | 系统拼接结果 |
|------|-------------|-------------|
| OpenAI | https://api.openai.com | https://api.openai.com/v1/chat/completions |
| Anthropic | https://api.anthropic.com | https://api.anthropic.com/v1/messages |
| Moonshot/Kimi | https://api.moonshot.cn | https://api.moonshot.cn/v1/chat/completions |
| 自定义 | 完整 URL | 不做拼接，原样使用 |

## 3. 非功能需求

### 3.1 性能需求

| 指标 | 要求 |
|------|------|
| 文件上传 | 支持最大 200MB 的 Word 文档 |
| 并发审查 | 支持同时运行 8 个审查任务（异步线程池） |
| 文档解析 | 按一级标题切分章节，每个切片最大 25600 tokens |
| AI 调用重试 | 单次调用失败自动重试，最多 3 次，间隔 5 秒 |
| 数据库连接池 | 最大 20 连接，最小空闲 5 连接 |

### 3.2 安全需求

| 项目 | 实现方式 |
|------|---------|
| 密码存储 | BCrypt 哈希加密（强度 10） |
| 接口认证 | JWT Token（HS256 签名） |
| API Key 保护 | 界面展示脱敏（前4后4位），编辑时不重输入则保持原值，禁止脱敏值覆盖真实密钥 |
| 会话管理 | 无状态设计，不使用服务器端 Session |
| 接口鉴权 | 基于角色的访问控制，方法级别注解 |
| CSRF | 已禁用（RESTful API + JWT 无需 CSRF） |

### 3.3 可用性需求

| 项目 | 要求 |
|------|------|
| 实时反馈 | WebSocket 实时推送审查进度和日志 |
| 错误处理 | API 统一错误响应格式，前端友好提示 |
| 任务管理 | 支持取消处理中的任务，避免资源浪费 |
| 容器健康检查 | 后端提供 /api/health 端点，Docker 定期检测 |
| 优雅停机 | Spring Boot 优雅关闭（5秒超时），Docker 10秒强制停止 |

### 3.4 部署需求

| 项目 | 要求 |
|------|------|
| 部署方式 | Docker Compose 一键部署 |
| 跨平台 | Mac、Windows、Linux 均可直接运行 |
| 行尾兼容 | .gitattributes 强制 LF，支持项目直接压缩拷贝到 Windows |
| 外部依赖 | 仅需安装 Docker Desktop，无需本地 Java/Node/PostgreSQL |
| 数据持久化 | PostgreSQL 数据和上传文件通过 Docker Volume 持久化 |
| 服务启动顺序 | PostgreSQL -> 后端（等待数据库健康） -> 前端（等待后端健康） |

## 4. 数据模型

### 4.1 实体关系

```
users 1---N rules                (创建者)
users 1---N rule_libraries       (创建者)
users 1---N scenarios            (创建者)
users 1---N review_tasks         (提交者)
users N---N rule_libraries       (通过 user_library_assignment 分配)

rule_libraries 1---N rules       (包含)
scenarios N---N rule_libraries   (通过 scenario_library_mapping 关联)
scenarios N---N rules            (通过 scenario_rule_mapping 关联)
scenarios 1---N review_tasks     (审查使用的场景)
```

## 7. 审查引擎设计

### 7.1 文档解析

- 支持 .doc（旧版 Word）和 .docx（新版 Word）格式
- 按**一级标题（Heading 1）**切分文档为多个章节
- 解析过程中**跳过图片**，仅提取文本内容
- 保留表格内容（转为纯文本格式）
- 无一级标题的文档整体作为一个章节处理

### 7.2 文档切片

- 每个章节独立作为一个切片
- 如果章节超过最大 Token 限制（25600 tokens），进行二次拆分
- 二次拆分按段落边界切分，避免截断句子
- Token 估算规则：CJK 字符 1 字 = 1 token，英文 4 字符 = 1 token

### 7.3 审查提示词构建

- 从审查场景中获取关联的所有规则库
- 加载规则库中的所有规则
- 将规则内容组合为系统提示词（System Prompt）
- 文档切片作为用户消息（User Message）发送给 AI

### 7.4 AI 调用流程

```
对每个文档切片：
  1. 构建请求（system prompt + chunk content）
  2. 发送 HTTP POST 到 AI 模型 API
  3. 如果失败，等待 5 秒后重试（最多 3 次）
  4. 解析 AI 响应（尝试 JSON 解析，失败则存为原始文本）
  5. 通过 WebSocket 推送进度更新
最后汇总所有切片的审查结果
```

### 7.5 结果存储

审查结果以 JSONB 格式存储在 `review_tasks.ai_result` 字段中，结构如下：

```json
{
  "totalChunks": 5,
  "overallScore": 82,
  "totalIssues": 12,
  "allIssues": [],
  "chunkResults": [
    {
      "chunk": 1,
      "chapterTitle": "第一章 概述",
      "totalChunks": 5,
      "estimatedTokens": 3200,
      "result": {}
    }
  ]
}
```

## 8. 实时通信设计

### 8.1 WebSocket 连接

- 端点：`/ws/task-progress`
- 认证：URL 查询参数携带 JWT Token
- 自动重连：连接断开后指数退避重连（最多 5 次）

### 8.2 消息格式

```json
{
  "taskId": "uuid-string",
  "status": "PROCESSING",
  "progress": 45,
  "message": "正在审查 [第三章 需求分析] (3/5，约3200 tokens)..."
}
```

### 8.3 推送时机

| 事件 | 进度值 | 消息内容 |
|------|--------|---------|
| 开始审查 | 5% | 开始文件审查... |
| 文档解析完成 | 15% | 文档解析完成，共识别 N 个章节 |
| 规则加载完成 | 25% | 已加载 N 条审查规则 |
| 文档切片完成 | 30% | 文档已切分为 N 个片段 |
| 每个切片审查中 | 30-90% | 正在审查 [章节名] (m/n) |
| 汇总结果 | 95% | 正在汇总审查结果... |
| 审查完成 | 100% | 审查完成 |
| 审查失败 | - | 错误原因 |
| 任务取消 | - | 任务已取消 |
