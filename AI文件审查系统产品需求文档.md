# AI 文件审查系统产品需求文档

**版本**：V1.1  
**文档状态**：基于当前项目实现反推更新  
**适用范围**：AI 文件审查系统 Web 端、后端服务、审查引擎、部署与运维能力

---

## 1. 产品概述

### 1.1 产品定位

AI 文件审查系统是一款面向企业、科研机构和工程项目团队的智能文档审查平台。系统以 Word 文档为审查对象，以规则库和审查场景为审查依据，接入多厂商大语言模型，对文件内容进行分章节、分规则、可追踪的自动化审查，输出问题清单、判定依据、修改建议和 Excel 审查意见表。

产品重点解决以下问题：

- 人工审查耗时长、审查标准不一致、结果难追踪。
- 专业规则数量多，规则与文档章节匹配成本高。
- 多模型能力差异大，缺少统一接入、测试和运行配置。
- 长文档审查需要切片、并发、重试、实时反馈和可导出结果。

### 1.2 目标用户

| 用户角色 | 核心诉求 | 典型使用场景 |
|---------|---------|-------------|
| 项目主管 | 团队、权限和审查资源统一治理 | 创建用户、分配角色、分配规则库访问权限、监督审查任务 |
| 管理员 | 配置规则、场景和模型 | 建立规则库、上传规则包、维护审查场景、配置 AI 模型 |
| 普通用户 | 快速完成文档审查 | 上传 Word 文件、选择审查场景与模型、查看结果、导出意见 |

### 1.3 产品目标

- 将 Word 文件审查流程从“人工逐条核查”升级为“规则驱动 + AI 辅助审查”。
- 支持复杂规则包拆解、元信息编辑、按章节命中规则，降低无关规则对模型上下文的占用。
- 支持 OpenAI 兼容接口、Moonshot/Kimi、Anthropic、阿里通义及自定义供应商等多模型接入。
- 通过实时日志、进度条、规则调度信息和 Excel 导出，形成可解释、可交付的审查闭环。
- 通过 Docker Compose 降低部署门槛，支持本地或内网环境快速运行。

## 3. 用户与权限需求

### 3.1 账号注册与登录

- 用户可通过邮箱、姓名和密码注册账号。
- 注册成功后默认角色为普通用户。
- 用户使用邮箱和密码登录，登录成功后获取 Access Token 与 Refresh Token。
- Access Token 默认有效期 1 小时，Refresh Token 默认有效期 7 天。
- 前端请求自动携带 Bearer Token；Token 过期后优先尝试刷新。
- 用户可退出登录，清理本地认证状态并返回登录页。

### 3.2 个人中心

- 用户可查看个人姓名、邮箱、角色等信息。
- 用户可修改密码，修改时必须输入原密码并校验通过。
- 密码在后端以 BCrypt 哈希方式存储。

### 3.3 角色权限

系统内置三级角色：普通用户（user）、管理员（admin）、项目主管（supervisor）。

| 功能项 | 普通用户 | 管理员 | 项目主管 |
|--------|---------|--------|---------|
| 登录、查看个人信息、修改密码 | Y | Y | Y |
| 发起文件审查 | Y | Y | Y |
| 查看自己的审查任务 | Y | Y | Y |
| 取消、删除、重新审查自己的任务 | Y | Y | Y |
| 查看审查详情和导出 Excel | Y | Y | Y |
| 查看已授权规则 | Y | Y | Y |
| 管理规则库和规则 | - | Y | Y |
| 管理审查场景 | - | Y | Y |
| 管理 AI 模型 | - | Y | Y |
| 管理用户 | - | - | Y |
| 为用户分配规则库 | - | - | Y |

### 3.4 用户管理

项目主管可进入用户管理页面：

- 分页查看所有用户。
- 创建用户并设置邮箱、姓名、密码和角色。
- 修改普通用户与管理员的角色。
- 删除用户账号。
- 为指定用户分配可访问的规则库。
- 查看指定用户已分配的规则库列表。

## 4. 工作台需求

### 4.1 任务统计

工作台顶部展示当前用户维度统计：

| 指标 | 说明 |
|------|------|
| 审查总数 | 当前用户创建的全部审查任务数 |
| 已完成 | 状态为 COMPLETED 的任务数 |
| 进行中 | 状态为 PROCESSING 的任务数 |
| 今日审查 | 当天创建的任务数 |

### 4.2 任务列表

- 分页展示当前用户任务，默认每页 10 条。
- 支持按任务状态筛选：待处理、处理中、已完成、失败、已取消。
- 列表字段包括文件名、模型、任务状态、问题数、创建时间、操作。
- 任务状态显示为彩色标签。
- 进行中的任务通过 WebSocket 推送触发列表刷新。

### 4.3 新建审查

新建审查使用弹窗流程：

1. 上传文件
   - 支持拖拽或点击选择。
   - 支持 `.doc`、`.docx` Word 文件。
   - 后端最大上传文件限制为 200MB；前端组件可按页面配置限制上传大小。
   - 文件选择后展示文件名并阻止浏览器自动上传。

2. 配置审查参数
   - 选择审查场景。
   - 选择已启用 AI 模型。
   - 提交后创建任务记录，并异步执行审查。
   - 上传请求超时时间应放宽到 120 秒，以适配较大文件或慢速网络。

### 4.4 任务操作

| 操作 | 可用状态 | 说明 |
|------|---------|------|
| 查看详情 | 处理中、已完成、失败、已取消 | 跳转到审查工作区 |
| 取消 | 待处理、处理中 | 将任务标记为 CANCELLED，并通知异步任务尽快停止 |
| 重新审查 | 已完成、失败、已取消 | 复用原文件、场景和模型创建新任务 |
| 删除 | 非处理中任务 | 删除任务记录，需二次确认 |

## 5. 审查工作区需求

### 5.1 页面布局

审查详情页采用左右分栏：

- 左侧：审查结果列表或原始 JSON 结果。
- 右侧：审查概要、规则调度、失败原因、进度条、实时日志。

### 5.2 审查结果展示

完成后展示所有汇总问题：

- 问题分类。
- 规则名称和规则编码。
- 问题描述。
- 问题位置或原文引用。
- 修改建议。
- 判定依据。
- 严重程度。

当结果无法提取为标准问题数组时，页面应回退展示完整 JSON，避免结果丢失。

### 5.3 审查概要

审查概要应展示：

- 综合评分（若模型返回）。
- 文档分块数。
- 发现问题数。
- 严重程度分布：高、中、低、未标注。
- 问题分类分布。
- 任务失败原因（若存在）。

### 5.4 规则调度展示

若后端返回 `chunkResults`，页面展示每个章节/切片命中的规则：

- 章节标题或片段序号。
- 命中规则数量。
- 前若干条命中规则名称标签，超出部分以数量折叠显示。

### 5.5 实时日志与进度

- 页面进入后建立 WebSocket 连接。
- 日志在全局布局层订阅并写入前端日志状态，用户离开详情页再返回时仍可看到历史日志。
- 日志包含时间、等级、消息和进度值。
- 日志等级包括信息、成功、警告、错误。
- 处理中任务显示进度条，进度由 WebSocket 消息驱动。
- 终态消息（COMPLETED、FAILED、CANCELLED）触发任务详情刷新。

### 5.6 Excel 导出

- 审查完成且存在结果时，用户可导出 Excel。
- 文件名格式：`审查意见_{原文件名或任务ID}.xlsx`。
- Excel Sheet 名称为“审查意见”。
- 导出列包括：序号、章节、严重程度、问题分类、规则编码、审查意见、判定依据、是否接受。
- 当 `chunkResults` 中没有可提取问题时，应回退从 `allIssues` 提取。

## 6. 规则库与规则管理需求

### 6.1 规则库管理

管理员和项目主管可维护规则库：

- 创建规则库：填写名称和描述。
- 查看规则库列表：展示规则库名称、描述、规则数量、更新时间。
- 编辑规则库：修改名称和描述。
- 删除规则库：级联删除库内规则及相关映射。
- 进入规则库后查看该库下规则列表。

### 6.2 规则上传

- 支持上传 `.md`、`.json`、`.txt`、`.yaml`、`.yml` 文件。
- 前端限制规则文件不超过 5MB。
- 上传时可指定所属规则库。
- 单个 JSON 规则包可被解析为多条独立规则。
- 规则上传后返回解析出的规则列表；若解析出多条规则，应提示解析数量。

### 6.3 多规则包解析

系统需要识别常见规则包结构：

- `section_prompts`。
- `rules`。
- `prompts`。
- 顶层键值为规则对象的 JSON。

对于每条子规则：

- 使用键名作为规则名称或规则编码。
- 将 `system_prompt` 和 `user_prompt` 渲染为可读 Markdown 内容。
- 若无法识别为多规则包，则作为单条规则保存。

### 6.4 规则元信息

规则支持以下元信息：

| 字段 | 说明 |
|------|------|
| 规则名称 | 页面展示名称 |
| 规则编码 | 标准或规则包中的唯一标识 |
| 规则类型 | global、section_specific、document_specific、output |
| 文档类型 | 适用文档类型 |
| 目标章节 | 标准章节号或文档章节号 |
| 适用范围 | 关键词或 Scope，用于匹配章节标题 |
| 严重程度 | high、medium、low |
| 规则描述 | 规则摘要 |
| 来源文件 | 上传文件名 |

元信息来源优先级：

1. 用户在界面手动编辑并保存的数据库字段。
2. 规则文件中的 YAML frontmatter 或 JSON 元信息。
3. 系统自动识别和启发式推断。

### 6.5 元信息自动识别

系统应支持：

- Markdown YAML frontmatter，例如 `rule_type`、`sections`、`keywords`。
- JSON 字段，例如 `type`、`rule_type`、`scope`、`Scope`、`sections`、`target_sections`。
- 中英文规则类型别名归一化：通用/global、专项/section_specific、文档级/document_specific、输出/output。
- DO-160/RTCA 相关内容识别：自动提取标准章节，如“DO-160G 第13章”“Section 13 of DO-160”“4.5.1条”。
- 环境试验类关键词识别，如温度、湿热、冲击、振动、霉菌、盐雾、防火等。

### 6.6 规则列表与操作

- 规则以规则库为入口分页展示。
- 支持查看规则预览。
- 预览时展示规则元信息和规则正文。
- 支持编辑规则元信息。
- 支持单条删除和批量删除。
- 普通用户仅可查看有权限的规则，不可上传、编辑或删除。

## 7. 审查场景需求

### 7.1 场景定义

审查场景是一组规则库的组合，用于定义一次审查所采用的规则范围。场景与规则库为多对多关系。

### 7.2 场景管理

管理员和项目主管可：

- 分页查看场景列表。
- 创建场景：填写名称、描述，并至少关联一个规则库。
- 编辑场景：修改名称、描述和关联规则库。
- 删除场景。
- 查看场景详情：展示场景基本信息和关联规则库。

### 7.3 场景使用

- 用户新建审查时必须选择场景。
- 审查引擎根据场景加载关联规则库下的有效规则。
- 普通用户的规则可见性和审查可用范围应受规则库分配影响。

## 8. AI 模型管理需求

### 8.1 模型列表

管理员和项目主管可分页管理模型配置：

- 模型名称。
- 供应商。
- 模型标识。
- API 地址。
- 最大 Token。
- Temperature。
- 思考模式。
- 启用状态。
- 编辑、删除、启停操作。

### 8.2 添加与编辑模型

| 字段 | 必填 | 说明 |
|------|------|------|
| 模型名称 | Y | 页面展示名称，需唯一 |
| 供应商 | Y | OpenAI、Anthropic、Moonshot、阿里通义、自定义 |
| 自定义供应商名称 | 条件必填 | 选择自定义供应商时填写 |
| 模型标识 | Y | 实际请求的模型 ID，如 `gpt-4o`、`kimi-k2.6` |
| API 地址 | Y | 可填写域名、基础地址或完整接口地址 |
| API Key | 新增必填 | 编辑时可留空或保留脱敏值，不覆盖原密钥 |
| 最大 Token | 可选 | 生成最大 Token；思考模式下后端最低提升到 16000 |
| Temperature | 可选 | 普通模型参与请求；思考模型请求时忽略 |
| 启用状态 | 可选 | 启用后可在新建审查时选择 |
| 思考模式 | 可选 | 适配推理/思考模型 |

### 8.3 API 地址补全

系统根据供应商补全请求地址：

| 供应商 | 补全逻辑 |
|------|---------|
| OpenAI | 基础地址补全 `/v1/chat/completions` |
| Anthropic | 基础地址补全 `/v1/messages` |
| Moonshot | 基础地址或 `/v1` 补全 `/chat/completions` |
| 阿里通义 | 保留用户地址 |
| 自定义 | 默认按 OpenAI 兼容接口补全 `/v1/chat/completions` |

### 8.4 思考模式

系统支持思考/推理模型特殊处理：

- 根据模型标识自动建议是否启用思考模式。
- 匹配示例：`kimi-k2-thinking`、`kimi-k2.6`、`glm-4.7`、`glm-5.1`、`deepseek-r1`、包含 `thinking` 或 `reasoner` 的模型。
- 思考模式下请求不发送 `temperature`，避免供应商固定温度导致请求失败。
- 思考模式下 `max_tokens` 最低提升至 16000，避免推理内容占用输出预算。
- 响应解析优先使用 `content`，若为空则回退使用 `reasoning_content`。

### 8.5 连接测试

模型配置弹窗内提供“测试连接”能力：

- 支持新增前测试，也支持编辑已有模型时复用已保存 API Key。
- 发送最小化测试请求，降低调用成本。
- 普通模型测试输出上限为 16 tokens；思考模型测试输出上限至少 16000 tokens。
- 成功时展示解析后的请求地址、响应耗时和模型回复片段。
- 失败时展示 HTTP 状态、响应体或配置错误提示。

### 8.6 密钥安全

- 后端返回模型配置时对 API Key 脱敏展示。
- 编辑模型时，如果 API Key 为空或包含脱敏占位符，则保持原密钥不变。
- 后端调用模型前校验 API Key，禁止使用脱敏值发起真实调用。

## 9. 审查引擎需求

### 9.1 审查主流程

```text
上传 Word 文件
  -> 创建审查任务
  -> 异步执行
  -> 解析文档章节
  -> 加载场景规则
  -> 准备规则元信息
  -> 文档章节切片
  -> 按章节标题分发规则
  -> 并发调用 AI 模型
  -> 汇总结果
  -> 存储 JSONB
  -> WebSocket 推送完成
```

### 9.2 Word 解析

- 支持 `.doc` 和 `.docx`。
- 使用 Apache POI 提取文本。
- 按一级标题切分为章节。
- 无一级标题时，全文作为一个章节。
- 表格内容转为文本保留。
- 图片内容跳过。

### 9.3 文档切片

- 优先以章节为切片单位。
- 空章节不应产生无效 AI 调用。
- 超过最大 Token 的章节按段落、句子继续拆分。
- Token 估算应兼容中文和英文：CJK 字符按约 1 字 1 token，英文按约 4 字符 1 token。
- 默认单切片最大 Token 为 25600。

### 9.4 规则分发

规则分发策略：

| 规则类型 | 分发方式 |
|---------|---------|
| global | 应用于每个章节切片 |
| output | 作为全局输出规范参与每个切片提示词 |
| section_specific | 仅当章节一级标题命中目标章节或关键词时应用 |
| document_specific | 不参与逐章节审查，在章节审查后作为全文汇总规则单独处理 |
| 未标注规则 | 默认按 global 处理，保证兼容旧规则库 |

专项规则匹配要求：

- 仅匹配章节一级标题，避免正文中图号、表号、数值误触发。
- 支持“第 N 章/节”“section/chapter/clause/part N”“N.x 子章节”“标题首 token 为 N”等形式。
- 支持关键词命中，如“霉菌”“振动”“温度”等。

### 9.5 跨章节参考

系统应识别当前章节中对其他章节的引用，例如：

- “见第 X 章”。
- “参见 4.5 条”。
- “see chapter/section/clause X”。

命中后在提示词中附加相关章节作为支撑上下文，提升跨章节一致性审查能力。

### 9.6 AI 调用

- 每个切片构建独立提示词：规则提示词 + 当前章节内容 + 支撑上下文。
- AI 请求采用 HTTP POST。
- 调用失败时自动重试，默认最多 5 次。
- 默认重试间隔为 6000 ms。
- 单个文档内切片支持并发审查，默认最大并发为 8。
- 任务取消后，不再提交新的切片审查；已发出的 HTTP 请求允许自然结束。

### 9.7 结果解析与汇总

系统需要尽量解析 AI 返回的 JSON：

- 成功解析时提取问题数组、评分、分类、严重程度等字段。
- 解析失败时保留原始文本，避免信息丢失。
- 最终汇总结果存储到 `review_tasks.ai_result`。

目标结果结构：

```json
{
  "totalChunks": 5,
  "overallScore": 82,
  "totalIssues": 12,
  "severityCounts": { "high": 2, "medium": 7, "low": 3 },
  "categoryCounts": { "格式问题": 3, "内容缺失": 9 },
  "allIssues": [],
  "chunkResults": [
    {
      "chunk": 1,
      "chapterTitle": "13 霉菌试验",
      "estimatedTokens": 3200,
      "appliedRules": ["G-13-fungus"],
      "matchTraces": [],
      "result": {}
    }
  ]
}
```

## 10. 实时通信需求

### 10.1 WebSocket 连接

- 端点：`/ws/task-progress`。
- 认证方式：URL 查询参数携带 JWT Token。
- 前端全局保持一个 WebSocket 实例，避免重复连接。
- 支持按任务 ID 订阅，也支持全局 `*` 订阅。
- 消息去重，避免同一帧被多次展示。
- 断线后指数退避重连，最多 5 次。

### 10.2 消息格式

```json
{
  "taskId": "uuid-string",
  "status": "PROCESSING",
  "progress": 45,
  "message": "正在审查 [13 霉菌试验] (3/8)...",
  "timestamp": "2026-05-15T10:00:00"
}
```

### 10.3 推送节点

| 节点 | 进度 | 说明 |
|------|------|------|
| 任务开始 | 5% | 创建并开始处理 |
| 文档解析完成 | 15% | 识别章节数量 |
| 规则加载完成 | 25% | 加载有效规则数量 |
| 切片完成 | 30% | 生成切片数量 |
| 切片审查中 | 30%-90% | 当前章节、序号和总数 |
| 汇总结果 | 95% | 聚合各切片结果 |
| 完成 | 100% | 状态 COMPLETED |
| 失败 | - | 状态 FAILED，附失败原因 |
| 取消 | - | 状态 CANCELLED |

## 14. 非功能需求

### 14.1 性能需求

| 指标 | 要求 |
|------|------|
| 上传文件大小 | 后端支持最大 200MB Word 文件 |
| 规则文件大小 | 前端限制最大 5MB |
| 异步任务线程池 | 核心线程 4，最大线程 8，队列 100 |
| 单文档切片并发 | 默认最大 8 个切片并发调用 AI |
| 单切片大小 | 默认最大 25600 tokens |
| AI 调用重试 | 默认最多 5 次，间隔 6000 ms |
| 数据库连接池 | 最大 20，最小空闲 5 |
| 上传请求超时 | 前端提交审查上传请求 120 秒 |

### 14.2 安全需求

| 项目 | 要求 |
|------|------|
| 密码存储 | BCrypt 哈希 |
| 接口认证 | JWT Bearer Token |
| 接口授权 | 基于角色和方法级权限控制 |
| API Key 展示 | 脱敏返回，避免明文暴露 |
| API Key 更新 | 空值或脱敏值不覆盖原密钥 |
| 会话状态 | 无状态，不依赖服务端 Session |
| CSRF | REST + JWT 场景下禁用 |
| 用户数据隔离 | 用户只能访问自己的审查任务 |

### 14.3 可用性需求

- 后端异常统一返回标准响应结构。
- 前端请求拦截器统一处理错误提示和 Token 刷新。
- WebSocket 中断后自动重连。
- 审查结果解析失败时保留原始输出。
- 任务取消后及时更新状态并推送消息。
- Docker 健康检查保障服务启动顺序。

### 14.4 兼容性需求

- 前端支持现代 Chromium、Edge、Firefox 浏览器。
- Docker 部署支持 Windows、macOS、Linux。
- 项目通过 `.gitattributes` 保持 LF 行尾，降低跨平台脚本问题。
- Windows 环境通过 `.env` 支持 Docker Compose 路径转换。

## 16. 验收标准

### 16.1 核心业务验收

- 用户可登录系统并进入工作台。
- 管理员可创建规则库、上传规则包，并看到多规则解析结果。
- 管理员可编辑规则元信息，专项规则可按章节标题命中。
- 管理员可创建场景并关联规则库。
- 管理员可配置模型、测试连接、启用模型。
- 普通用户可上传 Word 文件并选择场景、模型提交审查。
- 审查过程中页面可实时显示日志和进度。
- 审查完成后可查看问题、统计、规则调度，并导出 Excel。
- 用户可取消、删除、重新审查自己的任务。
- 项目主管可创建用户、修改角色、分配规则库。

### 16.2 技术验收

- 后端接口均返回统一响应结构。
- 任务结果写入 PostgreSQL JSONB 字段。
- WebSocket 消息包含任务 ID、状态、进度和消息。
- AI API Key 不在列表接口中明文返回。
- Docker Compose 可一键启动前端、后端和数据库。
- 服务健康检查可用于容器依赖编排。
