# 后端文档审查处理流程

## 概述

当后端接收到规则库和待审查文件后，系统按以下流程进行处理。整个审查过程是**异步执行**的，用户提交任务后立即返回，审查进度通过 **WebSocket** 实时推送到前端。

---

## 1. 任务提交阶段

**入口**：`ReviewController.executeReview()` → `ReviewService.submitReview()`

```
用户上传文件（Word .doc/.docx）
       ↓
  保存文件到本地磁盘（/app/uploads/documents/）
       ↓
  创建 ReviewTask 记录（状态：PENDING）
       ↓
  触发异步执行 → executeReviewAsync(taskId)
       ↓
  立即返回任务信息给前端
```

- 文件保存路径：`{documents-dir}/{UUID}_{原始文件名}`
- 任务ID：由 MyBatis-Plus 的 `ASSIGN_UUID` 策略自动生成

---

## 2. 异步审查流程

**入口**：`ReviewService.executeReviewAsync()` （标注 `@Async("reviewTaskExecutor")`）

### 步骤 2.1：解析文档结构

```
WordParser.parseChapters(filePath)
       ↓
  识别文档格式（.doc 或 .docx）
       ↓
  .docx → 按一级标题（Heading 1）切分章节
  .doc  → 整篇文档作为单个章节
```

**WordParser 的处理细节：**

| 元素类型 | 处理方式 |
|---------|---------|
| 一级标题（Heading 1） | 作为章节分隔点，每遇到一个新标题就开始新章节 |
| 二级~六级标题 | 保留为 Markdown 格式（`##` `###` 等） |
| 普通段落 | 提取文本内容 |
| 表格 | 转换为 HTML `<table>` 格式，保留行列结构 |
| 图片/绘图 | 跳过，不纳入审查内容 |

**输出**：`List<Chapter>`，每个 Chapter 包含 `title`（章节标题）和 `content`（章节正文）

### 步骤 2.2：加载审查规则

```
RuleService.getRulesByScenarioId(scenarioId)
       ↓
  通过 scenario_library_mapping 找到关联的规则库
       ↓
  从规则库中获取所有有效规则（is_valid = true）
       ↓
  提取每条规则的 content 字段
       ↓
  RuleParser.buildSystemPrompt(ruleContents) → 构建系统提示词
```

**规则到提示词的转换**：

规则支持两种格式：
- **Markdown (.md)**：直接作为提示词文本使用
- **JSON (.json)**：解析其中的 `name`、`description`、`criteria`、`scoring` 等字段，格式化为结构化文本

最终系统提示词结构：
```
你是一名专业的文档审查员...
【输出要求】JSON格式...
【注意事项】...
=== 审查规则 ===
--- 规则 1 ---
{规则1内容}
--- 规则 2 ---
{规则2内容}
```

### 步骤 2.3：文档切片

```
ChunkUtils.chunkByChapters(chapters, maxChunkTokens)
       ↓
  遍历每个章节：
    如果章节 token 数 ≤ maxChunkTokens(25600) → 整章作为一个切片
    如果章节 token 数 > maxChunkTokens → 按段落再切分
       ↓
  如果单个段落仍然超长 → 按句子切分
```

**Token 估算规则**：
- 中文字符：1 字符 ≈ 1 token
- 英文/数字：4 字符 ≈ 1 token

**输出**：`List<ChunkResult>`，每个包含 `label`（章节标题）、`content`（切片内容）、`estimatedTokens`

### 步骤 2.4：逐片段调用 AI 模型

```
for 每个切片 chunk:
    检查任务是否被取消 → 如果取消则终止
       ↓
    构建用户消息：
      "章节: {章节标题} ({序号}/{总数})\n\n{切片内容}"
       ↓
    callWithRetry(modelConfig, systemPrompt, userContent)
       ↓
    最多重试 3 次，每次间隔 5 秒
       ↓
    WebSocket 推送当前进度（30% ~ 90%）
```

**AI 调用细节**（`AiModelService.callAiModel()`）：

```
构建 OpenAI 兼容的请求体：
{
  "model": "{model_key}",
  "temperature": 0.1,
  "max_tokens": 4096,
  "messages": [
    {"role": "system", "content": "{系统提示词}"},
    {"role": "user", "content": "{切片内容}"}
  ]
}
       ↓
根据 provider 类型拼接完整 API URL：
  openai  → {endpoint}/v1/chat/completions
  custom  → {endpoint}（用户填写完整URL）
       ↓
发送 HTTP POST 请求
       ↓
解析响应 → choices[0].message.content
```

### 步骤 2.5：解析 AI 响应

```
AI 返回的文本
       ↓
去除可能的 markdown 代码块标记（```json ... ```）
       ↓
尝试解析为 JSON Map
       ↓
成功 → 存为结构化数据
失败 → 以原始文本存储
```

期望的 AI 返回 JSON 结构：
```json
{
  "overall_score": 85,
  "summary": "本章节审查总结",
  "issues": [
    {
      "severity": "high/medium/low",
      "location": "问题所在位置",
      "description": "问题描述",
      "suggestion": "修改建议",
      "rule": "对应规则名称"
    }
  ],
  "passed_items": ["通过的检查项"]
}
```

### 步骤 2.6：结果聚合

```
收集所有切片的审查结果
       ↓
计算平均评分（overall_score 的均值）
       ↓
汇总所有 issues 到 allIssues 列表
       ↓
统计问题总数 totalIssues
       ↓
组装最终结果：
{
  "totalChunks": N,
  "chunkResults": [...],
  "overallScore": 平均分,
  "totalIssues": 问题总数,
  "allIssues": [所有问题]
}
       ↓
保存到 review_tasks.ai_result（JSONB 字段）
       ↓
更新任务状态为 COMPLETED
       ↓
WebSocket 推送完成通知
```

## 3. 数据流向图

```
┌─────────┐     ┌───────────┐     ┌──────────────┐
│  规则库   │────→│  审查场景  │────→│ 场景-规则库映射 │
│(rules)   │     │(scenarios)│     │(scenario_    │
│          │     │           │     │ library_     │
│          │     │           │     │ mapping)     │
└─────────┘     └───────────┘     └──────────────┘
                                         │
                                         ↓
┌─────────┐     ┌───────────┐     ┌──────────────┐
│Word文档   │────→│ WordParser │────→│  ChunkUtils  │
│(.docx)   │     │ 解析+切章  │     │   切片处理    │
└─────────┘     └───────────┘     └──────────────┘
                                         │
                     ┌───────────┐       │
                     │RuleParser │       ↓
                     │构建提示词  │  ┌──────────────┐

                                    └──────┬───────┘
                                           ↓

                                    │  结果聚合     │
                                    │  存储到DB     │
                                    │  WebSocket通知│
                                    └──────────────┘
```

## 4. 关键配置项

| 配置项 | 路径 | 默认值 | 说明 |
|-------|------|-------|------|
| `review.chunk.max-tokens` | application.yml | 25600 | 每个切片最大 token 数 |
| `review.retry.max-attempts` | application.yml | 3 | AI 调用最大重试次数 |
| `review.retry.interval-ms` | application.yml | 5000 | 重试间隔（毫秒） |
| `file.documents-dir` | application.yml | ./uploads/documents | 文档存储目录 |
| `async.core-pool-size` | application.yml | 4 | 异步线程池核心线程数 |
| `async.max-pool-size` | application.yml | 8 | 异步线程池最大线程数 |

---

## 5. 异常处理

| 异常类型 | 处理方式 |
|---------|---------|
| 文档解析失败 | 标记任务为 FAILED，记录错误原因 |
| 规则加载为空 | 标记任务为 FAILED，提示未找到有效规则 |
| AI 调用超时/错误 | 重试最多 3 次，全部失败后标记 FAILED |
| 用户主动取消 | 在每个切片处理前检查取消标记，标记 CANCELLED |
| AI 返回非 JSON | 以原始文本存储，不影响整体流程 |

---

## 6. Excel 导出

审查完成后，用户可导出 Excel 审查意见表：

| 列名 | 内容来源 |
|------|---------|
| 序号 | 从 1 开始递增 |
| 章节 | chunkResults 中的 chapterTitle |
| 审查意见 | issues 中的 description + suggestion |
| 判定依据 | issues 中的 location + rule + severity |
| 是否接受 | 空白（供人工填写） |
