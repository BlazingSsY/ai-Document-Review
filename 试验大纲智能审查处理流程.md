# 试验大纲智能审查处理流程

## 一、定位与设计原则

本文档描述“试验大纲智能审查”如何嵌入当前 AI 文件审查系统的通用功能框架中。试验大纲不应被实现为后端硬编码的专用流程，而应作为一个可配置的审查场景运行：

- 通过“规则库”沉淀试验大纲通用规则、专项规则、输出格式约束。
- 通过“审查场景”组合一个或多个规则库，形成“试验大纲审查场景”。
- 通过现有审查任务流程完成文档上传、Word 解析、章节切片、AI 调用、结果汇总、结果导出。
- 后续新增其他文档审查场景时，只新增规则库、规则文件和场景配置；除非需要新的文档解析能力或新的规则调度能力，否则不修改主审查流程。

当前后端已具备的核心框架如下：

| 框架能力 | 当前实现 |
|---|---|
| 规则库 | `rule_libraries` 管理规则集合 |
| 审查规则 | `rules` 保存 `.md` / `.json` 规则原文和元数据 |
| 审查场景 | `scenarios` 表示一类审查业务 |
| 场景与规则库关联 | `scenario_library_mapping` 将多个规则库组合为一个场景 |
| 审查任务 | `review_tasks` 记录文件、场景、模型、状态和 AI 结果 |
| 规则加载 | `RuleMapper.findByScenarioId` 按场景加载关联规则库下的有效规则 |
| 文档解析 | `WordParser.parseChapters` 按一级标题解析章节 |
| 文档切片 | `ChunkUtils.chunkByChapters` 按章节和 Token 限制切片 |
| Prompt 构建 | `RuleParser.buildSystemPrompt` 将规则组合为系统提示词 |
| AI 调用 | `AiModelService.callAiModel` 调用所选模型并支持失败重试 |
| 结果存储 | `review_tasks.ai_result` 保存汇总 JSON，支持 Excel 导出 |

## 二、试验大纲场景配置模型

试验大纲审查应配置为一个独立场景，例如“试验大纲智能审查”。该场景关联若干规则库，每个规则库只表达一个清晰职责。

建议的规则库拆分：

| 规则库 | 作用 | 适用范围 |
|---|---|---|
| 试验大纲通用规则库 | 检查文档基础质量、格式、试验单位、试验场所、图表引用、术语一致性等 | 全文档或所有章节 |
| 试验大纲结构完整性规则库 | 检查试验目的、依据、试验件、测试设备、试验条件、试验步骤、判据、记录要求等是否完整 | 全文档或相关章节 |
| DO-160G 专项规则库 | 按 DO-160G 章节配置霉菌、磁影响、流体敏感性等专项审查规则 | 命中对应章节时重点适用 |
| 输出规范规则库 | 约束 AI 返回 JSON 字段、问题描述、修改建议、判定依据表达方式 | 所有切片 |

规则文件仍使用当前支持的 `.md` 或 `.json` 格式上传。为了便于扩展和后续自动调度，建议在规则内容开头加入统一元信息，但不要求立即改表：

```markdown
---
scenario: 试验大纲智能审查
rule_code: DO160G-13-001
rule_type: section_specific
applies_to:
  standard: DO-160G
  sections: ["13"]
  keywords: ["霉菌", "Fungus"]
severity: high
---

# 霉菌试验程序完整性检查

检查试验大纲中霉菌试验程序是否包含试验条件、持续时间、样品状态、判定依据和记录要求。
```

这些元信息在当前版本中会作为规则文本进入 Prompt，后续如需降低调用成本，可由后端解析为结构化规则元数据，用于规则筛选和任务编排。

## 三、当前可落地的审查处理流程

### 阶段 1：规则与场景准备

管理员或项目主管先在系统中完成配置：

1. 创建规则库，例如“试验大纲通用规则库”“DO-160G 专项规则库”。
2. 上传 Markdown 或 JSON 规则文件，系统通过 `RuleParser.validate` 校验格式，并将规则原文保存到 `rules.content`。
3. 创建“试验大纲智能审查”场景，并关联上述规则库。
4. 配置可用 AI 模型，确保用户提交审查时可选择已启用模型。

该阶段利用现有规则、规则库、场景管理能力完成，不需要为试验大纲新增专用接口。

### 阶段 2：用户提交审查任务

用户在工作台发起新建审查：

1. 上传 `.doc` 或 `.docx` 试验大纲文件。
2. 选择“试验大纲智能审查”场景。
3. 选择一个已启用 AI 模型。
4. 前端调用 `POST /api/v1/reviews/execute`，提交 `file`、`scenarioId`、`selectedModel`。
5. 后端 `ReviewService.submitReview` 保存文件，创建 `review_tasks` 记录，状态为 `PENDING`，随后异步执行审查。

此处的关键点是：审查任务只依赖场景 ID，不依赖“试验大纲”硬编码类型。未来新增“合同审查”“需求文档审查”等场景，也走相同入口。

### 阶段 3：文档解析与切片

异步任务启动后，后端执行统一文档处理：

1. 将任务状态更新为 `PROCESSING`。
2. 通过 WebSocket 推送“开始文件审查”。
3. 使用 `WordParser.parseChapters` 按一级标题解析 Word 文档。
4. 如果文档没有可解析内容，则任务失败并记录原因。
5. 使用 `ChunkUtils.chunkByChapters` 将章节转换为审查切片；章节过长时继续按 Token 限制拆分。
6. 将切片调试信息写入 `output/切片结果.json`，便于问题定位。

当前系统的切片单位是“章节切片”，适合试验大纲这类结构化文档。试验大纲规则应在 Prompt 中明确要求 AI 根据当前章节标题和内容判断适用性。

### 阶段 4：按场景加载规则

系统根据任务中的 `scenarioId` 加载规则：

1. `RuleService.getRulesByScenarioId` 调用 `RuleMapper.findByScenarioId`。
2. SQL 通过 `scenario_library_mapping` 找到场景关联的规则库。
3. 加载规则库下所有 `is_valid = true` 的规则。
4. `RuleParser.parseContent` 将 `.md` 或 `.json` 规则转换为可注入 Prompt 的文本。
5. `RuleParser.buildSystemPrompt` 组合输出格式要求、表格审查注意事项和全部规则内容。

当前版本是“场景级全量规则注入”：同一场景下的所有规则会进入每个切片的系统提示词。为了保证专项规则不会误判，试验大纲专项规则需要写明适用条件，例如“仅当当前章节为霉菌试验或明确涉及 DO-160G 第 13 章时执行本规则；否则在 passed_items 中说明不适用，不输出问题”。

### 阶段 5：逐切片 AI 审查

系统对每个切片执行统一 AI 调用：

1. 用户消息包含切片标签、序号、章节标题和章节正文。
2. 系统消息包含该场景加载出的全部规则。
3. AI 必须按统一 JSON 格式返回：

```json
{
  "summary": "本章节审查总结",
  "issues": [
    {
      "location": "问题所在位置",
      "description": "问题描述",
      "suggestion": "修改建议",
      "rule": "对应的审查规则名称"
    }
  ],
  "passed_items": ["通过或不适用的检查项"]
}
```

4. 单次调用失败时，`callWithRetry` 按配置重试，默认最多 3 次。
5. 每个切片完成后通过 WebSocket 推送进度。
6. 如果用户取消任务，后端在切片循环前检查取消标记并停止处理。

试验大纲规则需要把“通用审查”和“专项审查”的顺序写入规则内容中。例如：先检查文字、表格、引用、字段完整性，再判断是否命中某个专项试验章节并执行专项检查。

### 阶段 6：结果汇总与持久化

所有切片完成后，系统执行统一汇总：

1. `aggregateResults` 汇总每个切片返回结果。
2. 如果 AI 返回 `overall_score`，系统计算平均分。
3. 汇总所有 `issues` 到 `allIssues`。
4. 将结果保存到 `review_tasks.ai_result`。
5. 将完整审查结果写入 `output/审查结果.json`。
6. 更新任务状态为 `COMPLETED`，并通过 WebSocket 推送完成消息。

当前汇总结构如下：

```json
{
  "totalChunks": 5,
  "overallScore": 82,
  "totalIssues": 12,
  "allIssues": [],
  "chunkResults": [
    {
      "chunk": 1,
      "chapterTitle": "第一章 概述",
      "totalChunks": 5,
      "estimatedTokens": 3200,
      "result": {}
    }
  ]
}
```

### 阶段 7：结果查看与导出

用户在审查详情页查看任务进度、日志和结果。审查完成后可导出 Excel：

1. 前端调用 `GET /api/v1/reviews/tasks/{taskId}/export`。
2. 后端读取 `review_tasks.ai_result`。
3. 按切片和问题列表生成“审查意见”工作表。
4. 当前导出列为：序号、章节、审查意见、判定依据、是否接受。

试验大纲审查的报告字段应尽量映射到当前导出字段：问题描述和修改建议合并为“审查意见”，原文位置和规则名称合并为“判定依据”。

## 四、试验大纲专项规则的嵌入方式

### 1. 通用规则

通用规则应适用于所有切片，用于发现基础问题：

- 文档错别字、语序不当、语病。
- 试验单位、试验场所、试验日期、样品信息等基础字段缺失。
- 图号、表号、章节引用不一致。
- 表格单元格内容缺失、单位不统一、参数表达不完整。
- 试验前准备、试验前后功能性能检查、试验记录要求缺失。

通用规则可以放入“试验大纲通用规则库”，由场景全量加载。

### 2. 专项规则

专项规则面向具体试验类型或标准章节，例如 DO-160G 第 13 章霉菌试验、第 15 章磁影响试验等。由于当前后端尚未按规则元数据筛选切片，专项规则应采用“自判定适用性”的写法：

```markdown
# DO-160G 第 13 章霉菌试验审查规则

适用条件：
- 当前章节标题或正文明确出现“霉菌”“Fungus”“DO-160G 第13章”等内容时执行。
- 当前章节不涉及霉菌试验时，不输出 issue，只在 passed_items 中记录“本规则不适用于当前章节”。

审查要求：
1. 检查是否说明试验目的和适用标准。
2. 检查是否说明试验样品状态、试验周期、环境条件和菌种要求。
3. 检查是否给出试验前后检查项目和合格判据。
4. 发现缺失、矛盾或描述不明确时，输出 location、description、suggestion、rule。
```

这种写法能在不修改当前主流程的前提下，让试验大纲专项规则嵌入场景框架。

### 3. 输出规范规则

输出规范规则负责稳定 AI 结果结构，建议作为单独规则库关联到所有审查场景：

- 只输出 JSON，不输出 Markdown 代码块。
- 每条问题必须包含问题位置、问题描述、修改建议、对应规则。
- 未明确提及的内容不得主观补全，应判定为“未明确”并给出补充建议。
- 对不适用规则不得制造问题，应写入 `passed_items` 或在 summary 中说明。

## 五、面向扩展的目标流程

为了进一步减少无效模型调用，并支持更多文档场景，后续可在不破坏当前框架的基础上增强“规则调度层”。建议演进方向如下。

### 1. 规则元数据结构化

在 `rules` 表中增加可选元数据字段，或建立规则元数据表：

| 字段 | 说明 |
|---|---|
| rule_code | 规则编号，如 `DO160G-13-001` |
| rule_type | `global`、`section_specific`、`document_specific` |
| document_type | 适用文档类型，如 `test_outline`、`contract` |
| standard | 适用标准，如 `DO-160G` |
| target_sections | 适用标准章节或文档章节 |
| keywords | 章节匹配关键词 |
| severity | 默认风险等级 |
| enabled | 是否启用 |

有了结构化元数据后，系统可在 AI 调用前完成规则筛选，而不是依赖 AI 自行判断规则是否适用。

### 2. 场景处理策略配置

在 `scenarios` 增加可选策略配置，或建立场景策略表：

| 配置项 | 示例 | 作用 |
|---|---|---|
| document_type | `test_outline` | 标识场景对应文档类型 |
| parser_strategy | `word_heading_1` | 指定解析策略 |
| chunk_strategy | `chapter` | 指定切片策略 |
| rule_dispatch_strategy | `global_plus_matched_section` | 指定规则调度方式 |
| output_template | `review_excel_default` | 指定导出模板 |

新增文档审查场景时，通过配置选择策略，而不是复制一套后端流程。

### 3. 章节匹配层

针对试验大纲，可增加“文档章节与标准章节映射”步骤：

1. 从 `WordParser` 解析出的章节标题和正文摘要中提取关键词。
2. 使用规则元数据中的 `target_sections`、`keywords` 做确定性匹配。
3. 对低置信度章节，可调用轻量模型辅助判断。
4. 生成章节映射结果，例如：

| 文档章节 | 标题 | 标准 | 标准章节 | 置信度 |
|---|---|---|---|---|
| chunk-3 | 霉菌试验 | DO-160G | 13 | 0.98 |
| chunk-7 | 磁影响试验 | DO-160G | 15 | 0.97 |

该映射结果可用于后续规则筛选、审查报告分组和人工复核。

### 4. 规则调度层

目标调度方式：

1. 全局规则应用到所有切片。
2. 专项规则只应用到匹配章节。
3. 文档级规则可应用到全文摘要或汇总后的章节信息。
4. 输出规范规则始终注入。

调度后的 AI 调用可以从“每个切片携带全部规则”优化为“每个切片只携带相关规则”，减少 Token 消耗并降低误判概率。

### 5. 结果标准化增强

当前系统已经汇总 `issues`，后续可扩展问题字段：

| 字段 | 说明 |
|---|---|
| severity | 高 / 中 / 低 |
| category | 格式、完整性、标准符合性、逻辑一致性等 |
| rule_code | 命中的规则编号 |
| evidence | 判定依据 |
| acceptance_status | 是否接受，供人工处理 |

这些字段可以继续存入 `ai_result` JSONB，不影响现有任务模型。

## 六、新增其他文档审查场景的方法

当系统需要新增其他文档审查场景时，按以下顺序扩展：

1. 定义场景名称和审查边界，例如“合同审查”“需求规格说明书审查”“测试报告审查”。
2. 创建该场景对应的规则库，按“通用规则、专项规则、输出规范”拆分。
3. 上传规则文件，规则中写清适用条件、检查项、输出要求。
4. 创建审查场景并关联规则库。
5. 使用现有 `/api/v1/reviews/execute` 提交文档、场景和模型。
6. 如该文档类型不能用一级标题切分，再扩展新的 `parser_strategy` 或 `chunk_strategy`。
7. 如该场景存在大量专项规则，再扩展规则元数据和规则调度层。

这样可以保证新增场景优先通过配置完成，只有共性能力不足时才改后端。

## 七、当前实现与目标能力差异

| 能力 | 当前状态 | 建议处理 |
|---|---|---|
| 试验大纲作为独立场景 | 已支持，通过场景关联规则库实现 | 直接配置 |
| 通用规则全文审查 | 已支持，规则随场景注入每个切片 | 直接配置 |
| 专项规则按章节触发 | 部分支持，当前由规则文本引导 AI 自判定 | 后续增加规则元数据和调度层 |
| DO-160G 章节映射 | 尚未结构化实现 | 后续增加章节匹配层 |
| 多模型按规则调度 | 尚未实现，当前任务选择单一模型 | 后续在规则或场景策略中配置推荐模型 |
| 结果 JSON 汇总 | 已支持 | 可扩展 severity、rule_code 等字段 |
| Excel 导出 | 已支持基础列 | 可扩展模板和字段 |
| 新增审查场景 | 已支持，通过规则库和场景配置实现 | 保持主流程通用 |

## 八、结论

试验大纲智能审查应作为当前 AI 文件审查系统中的一个“可配置审查场景”落地，而不是新增一条专用后端流程。当前系统已经具备规则库、审查场景、规则加载、Word 章节切片、AI 审查、结果汇总和导出能力；试验大纲的业务差异应主要沉淀在规则库和规则内容中。

后续为了提升效率和可控性，可以逐步增加规则元数据、章节匹配层和规则调度层。该演进方式既能保留现有功能框架，又能支持未来持续新增其他文档审查场景。
