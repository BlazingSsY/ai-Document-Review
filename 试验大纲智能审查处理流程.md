# 试验大纲智能审查处理流程

> **阅读前须知（管线现状）**：本文写作时系统只有一条"逐章节"审查管线。当前系统前端开放两条物理隔离、可切换进入的审查管线：
> - **全文逐章审查（CHUNK）**——本文描述的流程（`ReviewService` + `RuleDispatcher` 章节调度），仍然有效，适合规则即提示词的通用审查。
> - **结构化精准审查（SAR）**——三路（结构+词法+语义）定位预期区域、区域级取证、清单式缺失检测、自适应复核、跨章一致性。由 `SarReviewService` 实现，检查单导入在 **`POST /api/v1/sar/rules/import-checklist`**。设计与调参见 [SAR_PIPELINE.md](SAR_PIPELINE.md)。
>
> RAG 相关文档保留为历史演进和技术背景，当前可操作入口以 CHUNK / SAR 为准。
>
> 另外，判定枚举已从早期设想的五级收敛为 **三级：Pass / Fail / Review**（Partial、N/A 并入 Review），且**输出不再有 severity 字段**——下文凡提到 severity / 五级处均以此为准。

## 一、定位与设计原则

本文档描述“试验大纲智能审查”如何嵌入当前 AI 文件审查系统的通用功能框架中。试验大纲不应被实现为后端硬编码的专用流程，而应作为一个可配置的审查场景运行：

- 通过“规则库”沉淀试验大纲通用规则、专项规则、输出格式约束。
- 通过“审查场景”组合一个或多个规则库，形成“试验大纲审查场景”。
- 通过现有审查任务流程完成文档上传、Word 解析、章节切片、AI 调用、结果汇总、结果导出。
- 后续新增其他文档审查场景时，只新增规则库、规则文件和场景配置；除非需要新的文档解析能力或新的规则调度能力，否则不修改主审查流程。

当前后端已具备的核心框架如下：

| 框架能力 | 当前实现 |
|---|---|
| 规则库 | `rule_libraries` 管理规则集合 |
| 审查规则 | `rules` 保存 `.md` / `.json` 规则原文和元数据（`rule_code`、`rule_type`、`sections`、`keywords`） |
| 审查场景 | `scenarios` 表示一类审查业务 |
| 场景与规则库关联 | `scenario_library_mapping` 将多个规则库组合为一个场景 |
| 审查任务 | `review_tasks` 记录文件、场景、模型、状态和 AI 结果 |
| 规则加载 | `RuleMapper.findByScenarioId` 按场景加载关联规则库下的有效规则 |
| 文档解析 | `WordParser.parseChapters` 按一级标题解析章节 |
| 文档切片 | `ChunkUtils.chunkByChapters` 按章节和 Token 限制切片 |
| 规则调度 | `RuleDispatcher.dispatchForChunk` 按规则类型调度：全局、一级标题命中、试验项目章节、文档级 |
| 规则元数据 | `MultiRuleParser` 解析当前 Markdown 规则块元数据；`RuleMetadata` 兼容 YAML frontmatter 或 JSON 顶层字段 |
| 逐章节审查 | 每个章节切片 = 一次 AI 调用；`reviewSingleChunk` 装配「命中规则 + 本章节切片 + 引用到的其他章节切片」 |
| 跨章节引用 | `ChapterReferenceResolver` 识别"见第X章/参见 4.5 条/参考<标题>"并把被引用章节作为上下文附带 |
| Prompt 构建 | `RuleParser.buildStructuredSystemPrompt` 四段式结构（ROLE / Schema / Few-shot / 规则清单） |
| 输出 Schema | `ReviewResultSchema.schema()`，集中维护 category / check_status 枚举 |
| 结构化输出 | OpenAI 兼容协议 `response_format=json_schema`，Anthropic `tool_use+tool_choice` |
| AI 调用 | `AiModelService.callAiModel(config, sys, user, AiCallOptions)`，统一 temperature/top_p/seed/max_tokens |
| 失败隔离与重审 | 单章节调用失败只标记 `failed/retryable`，不放大重试；可在结果页"重审失败切片"补审 |
| 结果存储 | `review_tasks.ai_result` 保存汇总 JSON（跨切片去重 + 枚举归一化），支持 Excel 导出 |

## 二、试验大纲场景配置模型

试验大纲审查应配置为一个独立场景，例如“试验大纲智能审查”。该场景关联若干规则库，每个规则库只表达一个清晰职责。

建议的规则库拆分：

| 规则库 | 作用 | 适用范围 |
|---|---|---|
| 试验大纲通用规则库 | 检查文档基础质量、格式、试验单位、试验场所、图表引用、术语一致性等 | 全文档或所有章节 |
| 试验大纲结构完整性规则库 | 检查试验目的、依据、试验件、测试设备、试验条件、试验步骤、判据、记录要求等是否完整 | 全文档或相关章节 |
| DO-160G 专项规则库 | 按 DO-160G 章节配置霉菌、磁影响、流体敏感性等专项审查规则 | 命中对应章节时重点适用 |
| 输出规范规则库 | 可选；输出格式已由 schema 强制，仅当需要补充全局措辞约定时使用 | 所有切片 |

规则文件用 `.md` 或 `.json` 格式上传。当前 DO-160G/QTP 章节规则主要采用 **Markdown 多规则模板**：一个 `.md` 文件内用二级标题 `##` 分隔多条规则，每条规则在标题下直接写元数据行。上传时 `MultiRuleParser` 会把每个 `##` 规则块拆成一条数据库规则；规则正文原样进入 prompt，不再从 Markdown 审查步骤中拆原子检查项。

当前推荐 Markdown 规则写法：

```markdown
## 1. 霉菌试验-设备名称与件号

- 规则编号：13-02-deviceIdentification

- 规则类型：section_specific

- 检查项：设备名称、件号

- 关键词：霉菌

### 审查内容

霉菌试验章节：核查“设备名称与件号”是否满足检查单确认目标。结论只允许三类：【通过】【不通过】【证据不足，待复核】。

### 审查步骤

1. 核查供应商件号必填合规性

...
```

识别条件：文件中同时存在 `### 审查步骤` 和 `- 规则类型：...` 元数据行时，系统按 `##` 标题拆分；没有 `规则编号` 的块不会入库。元数据行支持当前非加粗写法，也兼容旧的 `- **规则编号**：...` 写法。

YAML frontmatter / JSON 顶层字段仍然兼容，适合单条规则或标准 Rule JSON 包：

```markdown
---
scenario: 试验大纲智能审查
rule_code: DO160G-13-001
rule_type: section_specific
applies_to:
  standard: DO-160G
  sections: ["13"]
  keywords: ["霉菌", "Fungus"]
severity: high
---

# 霉菌试验程序完整性检查

检查试验大纲中霉菌试验程序是否包含试验条件、持续时间、样品状态、判定依据和记录要求。
```

元数据字段语义：

| 字段 | 作用 |
|---|---|
| `rule_code` | 全局唯一编号，模型 issue 必须从本批 manifest 中选 |
| `rule_type` | `global` / `section_specific` / `test_item_chapter` / `document_specific` / `output`，决定调度策略 |
| `applies_to.sections` 或 `- 目标章节` | `section_specific` 使用；命中一级标题开头章节号（如 `"13"`、`"13.5"`）时注入 |
| `applies_to.keywords` 或 `- 关键词` | `section_specific` 使用；命中一级章节标题关键词（"霉菌"、"Fungus"）时注入 |
| `severity` | 仅作人类可读备注，**不进入输出、不参与调度**（输出无 severity 字段） |

Markdown 元数据在上传时写入 `rules.rule_code/rule_type/keywords/...`；frontmatter 在审查前由 `RuleDispatcher.prepare` 解析。两种来源都会被 DB 列覆盖（便于用户在 UI 上编辑）。规则正文剥离 frontmatter 后再进 prompt；Markdown 多规则模板的标题行和 `---` 分隔线不会进入每条规则正文。当前 UI 支持编辑规则名称、编号、类型、适用章节、关键词和描述；仅保存这些元信息时不会刷新规则的排序时间，因此规则在列表中的位置保持不变。

`rule_type` 的实际应用范围：

| 类型 | 实际调度逻辑 |
|---|---|
| `global` / `output` | 注入每个章节切片 |
| `section_specific` | 只匹配一级章节标题；标题命中 `sections` 或 `keywords` 任一项才注入，不搜索正文、二级标题、三级标题 |
| `test_item_chapter` | 不按关键词触发；系统从“试验概述/试验项目概述”提取声明的试验项目清单，再把规则注入识别为试验项目的一级章节 |
| `document_specific` | 不参与逐章节调用；所有章节完成后单独跑一次文档级综合审查 |

规则列表页“适用范围”目前显示的是 `keywords` 元数据。对 `section_specific` 它就是一级标题触发词；对 `test_item_chapter` 或 `global`，这些词只是主题标签，真实应用范围由 `rule_type` 决定。

## 三、当前可落地的审查处理流程

### 阶段 1：规则与场景准备

管理员或项目主管先在系统中完成配置：

1. 创建规则库，例如“试验大纲通用规则库”“DO-160G 专项规则库”。
2. 上传 Markdown 或 JSON 规则文件，系统先通过 `RuleParser.validate` 校验格式，再由 `MultiRuleParser` 拆分：
   - 当前 Markdown 多规则模板按 `##` 拆成多条规则，每条规则写入一行 `rules`，`checks` 为空，审查时每条规则整体给模型判定一次；
   - 标准 Rule JSON / Excel 检查单生成的 JSON 可写入 `rule_checks` 原子检查项；
   - 其他单条 Markdown / JSON 保持一文件一规则。
3. 重复上传规则文件时，系统按同一规则库/文件夹内的 `rule_code` 优先、规则名称兜底自动合并：命中的旧规则会被新规则覆盖，文件中新增加的规则直接追加到库中。
4. 创建“试验大纲智能审查”场景，并关联上述规则库。
5. 配置可用 AI 模型，确保用户提交审查时可选择已启用模型。

该阶段利用现有规则、规则库、场景管理能力完成，不需要为试验大纲新增专用接口。

### 阶段 2：用户提交审查任务

用户在工作台发起新建审查：

1. 上传 `.doc` 或 `.docx` 试验大纲文件。
2. 选择“试验大纲智能审查”场景。
3. 选择一个已启用 AI 模型。
4. 前端调用 `POST /api/v1/reviews/execute`，提交 `file`、`scenarioId`、`selectedModel`。
5. 后端 `ReviewService.submitReview` 保存文件，创建 `review_tasks` 记录，状态为 `PENDING`，随后异步执行审查。

此处的关键点是：审查任务只依赖场景 ID，不依赖“试验大纲”硬编码类型。未来新增“合同审查”“需求文档审查”等场景，也走相同入口。

### 阶段 3：文档解析与切片

异步任务启动后，后端执行统一文档处理：

1. 将任务状态更新为 `PROCESSING`。
2. 通过 WebSocket 推送“开始文件审查”。
3. 使用 `WordParser.parseChapters` 按一级标题解析 Word 文档。
4. 如果文档没有可解析内容，则任务失败并记录原因。
5. 使用 `ChunkUtils.chunkByChapters` 将章节转换为审查切片；章节过长时继续按 Token 限制拆分。
6. 将切片调试信息写入 `output/切片结果_<file>_<model>_<ts>.json`，便于问题定位。

当前系统的切片单位是“章节切片”，适合试验大纲这类结构化文档。规则是否注入主要由 `RuleDispatcher` 在 AI 调用前判定；模型只审查本次已经注入的规则，不再承担大范围规则适用性筛选。

### 阶段 4：规则加载与按章节调度

系统根据任务中的 `scenarioId` 加载规则，并为每个切片单独选出适用规则：

1. `RuleService.getRulesByScenarioId` 调用 `RuleMapper.findByScenarioId`，按 `scenario_library_mapping` 找到场景关联的规则库，加载库下所有 `is_valid = true` 的规则。
2. `RuleDispatcher.prepare` 为每条规则解析 `RuleMetadata`（`rule_code`、`rule_type`、`sections`、`keywords`），并把 frontmatter 从规则正文中剥离；DB 字段优先于 frontmatter，保证 UI 编辑生效。
3. `RuleDispatcher.dispatchForChunk` 为每个切片做规则筛选：
   - 全局/输出类规则（`rule_type=global|output`）注入每个切片；
   - 专项规则（`rule_type=section_specific`）只在切片一级标题命中 `sections` 或 `keywords` 时注入，不匹配正文、二级标题或三级标题；
   - 试验项目章节规则（`rule_type=test_item_chapter`）只注入系统从“试验概述/试验项目概述”识别出的试验项目章节；该类型不按 `keywords` 触发；
   - 文档级规则（`rule_type=document_specific`）跳过逐切片，留到阶段 7 单独跑一次综合审查。
4. 调度前会调用 `extractDeclaredTestItems`：优先读取标题含“试验概述/试验项目概述”的章节；否则回退到正文含“试验项目有/鉴定试验项目有”的章节，从散文和试验项目一览表中提取项目名，供 `test_item_chapter` 使用。
5. 调度结果写入 `output/切片结果_<file>_<model>_<ts>.json`，方便排查“该切片为什么用/没用哪条规则”。

试验大纲专项规则应在元数据里写清楚 `sections` / `keywords`：例如 DO-160G 第 13 章霉菌试验规则 `keywords: ["霉菌"]` + `sections: ["13"]`，系统就会只在霉菌相关一级章节注入它，不再让模型自判定适用性。若章节标题可能写成“温度和高度试验”，关键词应包含“温度和高度”和“温度和高度试验”这类标题同义写法。

### 阶段 5：逐章节并行审查

系统以**章节切片**为最小单元，每个切片对应**一次独立 AI 调用**（调用量与章节数 1:1），并发由父线程 `Semaphore(review.parallel.chunk-concurrency)` 控制，提交到 `chunkReviewExecutor` 线程池。每个切片由 `reviewSingleChunk` 处理：

1. **组装 system prompt**：`RuleParser.buildStructuredSystemPrompt`，四段式结构（ROLE / Schema / Few-shot 锚点 / 规则清单按 `rule_code` 升序 + `[R-XXX]` 编号 + manifest）。规则只取 `RuleDispatcher` 为该章节命中的那几条。
2. **组装 user message**：章节标题 + 本章节切片正文 + 该章节通过 `ChapterReferenceResolver` 识别到的、被引用的其他章节切片（"见第X章 / 参见 4.5 条 / 参考<标题>"）。被引用章节用 `=== 以下为本章节引用的其他章节内容（仅供上下文参考…） ===` 包裹，明确告诉模型不要在被引用章节上套用本次规则。
3. **强制结构化输出**：
   - OpenAI / Moonshot / DeepSeek / GLM 等兼容协议 → `response_format={"type":"json_schema","json_schema":{...schema...}}`；
   - Anthropic → `tools=[submit_review_result]` + `tool_choice={"type":"tool"}`。
4. **统一参数**：`temperature=0`、`top_p=1`、`max_tokens=8192`、`seed=sha1(taskId+chunkIdx+0)` 取前 8 字节。每章节只调用一次（不再双采样）。
5. **强制覆盖检查项**：`RuleParser.expectedCheckCount` 会按“每条上传业务规则至少 1 条；有原子检查项则按原子检查项数量”计算 `check_results` 下限，并把该下限传给结构化输出 schema。无原子检查项的 Markdown 规则默认用 `规则编号-C001` 作为检查项编号。内置 `R-Q` 基础文字质量规则不计入这个下限；若无文字质量或图表编号问题，其 Pass 项不会进入检查项矩阵。
6. **解析与整理**：模型返回单切片 `{"summary":...,"issues":[...],"passed_items":[...],"check_results":[...]}`；解析失败会更换 seed 重试解析，仍失败时包装为一条"原始文本 issue"并保留，不丢结果。issue 现场补算指纹 `fingerprint = sha1(归一化location + "|" + rule_code)`，`confidence` 默认 `single`。
7. **规则快照补齐与漏项兜底**：`ensureBusinessRuleCoverage` 会为模型漏返的上传业务规则补 Review，`enrichCheckResultsWithRuleSnapshot` 会把本次注入规则/原子检查项的 `ruleName`、`rule_code`、`check_question`、`passCriteria` 补到 `check_results`，前端检查矩阵即使模型漏写规则名，也能用规则编号或检查项编号兜底展示。
8. **全文质量检查开关**：基础文字质量规则默认注入每章，包含错别字/语句/术语一致性，以及图号、表号唯一性、引用一致性、顺序和引用完整性检查。该内置规则只在 Fail/Review 时进入矩阵；若用户在新建审查时关闭全文质量检查，且某章节没有命中任何业务规则，该章节直接生成 `skipped` 结果，不调用模型。
9. **失败隔离**：单章节 AI 调用失败（如 HTTP 429）只把该切片标记为 `failed/retryable` 写入结果，**不做任何"拆批/翻倍重发"的放大动作**；其余章节照常推进。WebSocket 推送形如 `章节审查完成 5/40 [5 试验设备]，规则=3，耗时=820ms (5/40)`。

> **为什么不再批处理 / 双采样**：旧版用 `ChunkBatchPlanner` 把同签名切片打包成批、非思维模型对每片双采样、再用校准波动态调整 chunk 上限；批输出缺 `chunk_id` 或整批失败时会拆回单切片、每片再双采样重发。这套逻辑在模型限流（429）时会瞬间放大调用量、形成"越限流越猛打"的失败风暴，是"大量切片审查失败"的根因。现已下线，相关类（`ChunkBatchPlanner` / `ModelTier` / `buildBatchStructuredSystemPrompt` / `batchSchema`）暂保留但不再被调用。限流改由 `callWithRetry` 在 429 时尊重 `Retry-After` 头 + 更陡退避（封顶 60s）来吸收。

### 阶段 6：文档级综合审查 + 跨切片汇总

所有章节审查完成后：

1. **文档级 pass**：如果场景中有 `rule_type=document_specific` 规则，`buildDocumentLevelInput` 把章节目录 + 各切片摘要拼起来，跑一次综合 AI 调用（同样走收敛参数 + 结构化输出）。
2. **跨切片汇总（`aggregateResults`）**：
   - 按 fingerprint 跨切片去重（同一问题在多切片重复出现合并为一条，`occurrences` 记次数）；跨切片重复命中时把 `confidence` 由 `single` 升为 `high`；
   - `category` 强制映射到 {`格式`/`完整性`/`标准符合性`/`逻辑一致性`/`术语一致性`/`其他`}，未命中归 `其他`；
   - `passed_items` 按 `[R-XXX]` 抽出 rule_code 统计覆盖率 → `passedRuleCoverage`；
   - 冻结 `crossModelEligible` / `samplingStrategy`(=`single`) / `modelName` / `modelKey` 到结果根。
3. **持久化**：`review_tasks.ai_result` 与 `output/审查结果_<file>_<model>_<ts>.json` 同时落库与落盘。
4. **任务收尾**：状态置为 `COMPLETED`，WebSocket 推送完成消息（含失败切片数提示）。

### 阶段 7：结果查看与导出

用户在审查详情页查看任务进度、日志和结果。审查完成后可导出（`ReviewExportUtil`，CHUNK / SAR 共用）：

1. **检查项判定矩阵 Excel**：`GET /api/v1/reviews/tasks/{taskId}/export`。当 `ai_result` 含 `allCheckResults` 时（当前主路径），导出 12 列：**序号、章节、检查项编号、规则编码、判定、检查项、判定理由、证据、缺失项、建议、置信度、人工复核**；判定为三级 Pass/Fail/Review。
2. **旧问题列表 Excel**（历史任务无 `allCheckResults` 时回退）：导出 7 列：**序号、章节、问题分类、规则编码、审查意见、判定依据、是否接受**（无"严重程度"列）。
3. **Word 审查报告**：`GET /api/v1/reviews/tasks/{taskId}/report`，`.docx`，含任务信息、概要、检查项矩阵（序号/章节/检查项/系统判定/人工判定/理由/建议）与审计日志。
4. **审计 JSON**：`GET /api/v1/reviews/tasks/{taskId}/audit/export`。

`location` 字段用于"章节"列；导出取 `manualStatus`（人工改判后）优先于系统 `status`。SAR 管线在 `/api/v1/sar/reviews/tasks/{taskId}/export` 等同构路径下提供相同导出。

## 四、试验大纲专项规则的嵌入方式

### 1. 通用规则

通用规则应适用于所有切片，用于发现基础问题：

- 文档错别字、语序不当、语病。
- 试验单位、试验场所、试验日期、样品信息等基础字段缺失。
- 图号、表号、章节引用不一致。
- 表格单元格内容缺失、单位不统一、参数表达不完整。
- 试验前准备、试验前后功能性能检查、试验记录要求缺失。

通用规则可以放入“试验大纲通用规则库”，但应按实际应用方式选择 `rule_type`：

- 需要每章都审的规则写 `global` / `output`；
- 只对试验项目章节审的通用检查（如试验设备、辅助设备、试验条件、试验前检查）写 `test_item_chapter`；
- 只对某个固定章节审的通用检查（如“试验承试单位”章节）写 `section_specific` 并配置能命中一级标题的关键词。

### 2. 专项规则

专项规则面向具体试验类型或标准章节，例如 DO-160G 第 13 章霉菌试验、第 15 章磁影响试验。**专项规则必须写明规则编号、规则类型、检查项、关键词等元数据**，让 `RuleDispatcher` 在 AI 调用前就把它们筛掉不相关的切片，避免无效 token 消耗：

```markdown
## 1. 霉菌试验-试验程序

- 规则编号：13-08-testProcedure

- 规则类型：section_specific

- 检查项：试验程序

- 关键词：霉菌

### 审查内容

霉菌试验章节：核查“试验程序”是否满足检查单确认目标。结论只允许三类：【通过】【不通过】【证据不足，待复核】。

### 审查步骤

1. 核查试验程序完整性

检查 QTP 是否说明试验样品状态、试验周期、环境条件、菌种要求、判定依据和记录要求。
```

- `规则类型：section_specific` + `关键词` / `目标章节` 是命中调度的核心；命中后规则才会进入对应切片的 system prompt。
- `rule_code` 必填且全局唯一：模型只能从本批注入的 manifest 中选 `rule_code` 写到 issue 里，超出 manifest 的编号会被丢弃。
- 输出判定为三级 `status`（Pass/Fail/Review）+ `category` + `confidence`，**不再有 severity 字段**；若旧规则文件里仍有 `severity`，仅作人类备注，不进入输出、不参与调度。

> 旧文档里"专项规则自判定适用性"的写法已不再需要——`RuleDispatcher` 会做真正的筛选，规则正文里不必再重复"不适用就写入 passed_items"。

### 3. 输出规范规则

输出规范由 `ReviewResultSchema` 集中维护，已通过 provider 级 `response_format=json_schema` / `tool_use` 强制约束，**不再需要规则文件写输出格式说明**：

- 模型在解码阶段就被强制只能输出符合 schema 的 JSON 对象（OpenAI 兼容协议）或 tool_use input（Anthropic）；
- check_status 枚举：`Pass` / `Fail` / `Review`（三级；旧 Partial、N/A 已并入 Review）；
- category 枚举：`格式` / `完整性` / `标准符合性` / `逻辑一致性` / `术语一致性` / `其他`，无法归类 → `其他`；
- confidence 枚举：`high` / `medium` / `low` / `needs_review`；
- 不在 manifest 中的 `rule_code` 不允许出现；
- location 必须按 `一级 > 二级 > 三级` 格式逐字写原文。

如果你仍想在规则库中维护"通用输出说明"，可放入名称类似"试验大纲输出规范"的 `rule_type: output` 规则——它会作为全局规则注入每批 system prompt，但建议保持精简，避免和 schema 重复造成冗余 token。

## 五、面向扩展的演进方向

经过收敛 + 逐章节改造后，"规则元数据结构化"、"规则调度层"、"跨章节引用上下文"等原计划演进项已经实现。剩余演进方向集中在以下几点。

### 1. Excel 检查单 → SAR 原子检查项导入（已实现，持续增强）

DO-160G 等标准检查单常以 Excel 表格形式提供（每行 = 一条检查项）。当前 SAR 管线已提供 `POST /api/v1/sar/rules/import-checklist`，把 Excel 检查单导入为标准化规则 JSON，并写入 `sar_rules` / `sar_rule_checks`。接口支持 `libraryId` 和 `folderId`；重复导入时同一库/文件夹内同 `rule_code` 或同名规则会被覆盖，新检查项规则会直接追加。

形态识别（视 Excel 结构）：
- 扁平表（一行 = 一条检查项）
- 层级表（多级表头 + 合并单元格）
- 矩阵表（横轴章节、纵轴检查维度）

后续增强重点是导入前预览确认、专家在线编辑、标题列与字段映射配置，以及更复杂合并单元格的展开规则。

### 2. 场景处理策略配置（按需）

在 `scenarios` 增加可选策略配置表：

| 配置项 | 示例 | 作用 |
|---|---|---|
| document_type | `test_outline` | 标识场景对应文档类型 |
| parser_strategy | `word_heading_1` | 指定解析策略 |
| chunk_strategy | `chapter` | 指定切片策略 |
| rule_dispatch_strategy | `global_plus_matched_section` | 指定规则调度方式（当前是 hardcoded） |
| output_template | `review_excel_default` | 指定导出模板 |

新增审查场景时通过配置切策略，不必复制后端流程。

### 3. 单任务章节并发自适应（按需）

当前单任务的章节并发是固定值 `review.parallel.chunk-concurrency`（默认 6）。如果接入的模型限流（RPM/并发配额）差异较大，可按模型自适应并发：

1. 在 `ai_model_config` 表新增 `max_concurrency` 字段（可空）；
2. 前端模型管理页加输入框；
3. `ReviewService` 取该任务模型的 `max_concurrency`，未配置回退到全局默认。

> 注：旧版按 `ModelTier` 档位调批处理 `user_budget` / `max_chunks_per_batch` 的方案已随批处理一同下线；当前每章节单次调用，无需档位装箱。

### 4. 章节匹配层（增强专项规则调度精度）

当前 `RuleDispatcher.dispatchForChunk` 只匹配章节一级标题。如果文档章节命名与标准章节差异较大（例如自定义试验流程），可增加章节映射步骤：

1. 从 `WordParser` 解析出的章节标题和正文摘要中提取关键词；
2. 用规则元数据 `sections` / `keywords` 做语义匹配（可调用轻量模型辅助）；
3. 生成映射后再驱动调度。

示例：

| 文档章节 | 标题 | 标准 | 标准章节 | 置信度 |
|---|---|---|---|---|
| chunk-3 | 霉菌试验 | DO-160G | 13 | 0.98 |
| chunk-7 | 磁影响试验 | DO-160G | 15 | 0.97 |

### 5. 多模型对比 UI

后端已经在 `ai_result` 上冻结 `crossModelEligible` / `modelName` / `samplingStrategy` / `fingerprint`，前端可基于这些字段提供：

- "同一文档跑多个非思维模型，按 fingerprint 横向对比 issue"
- "思维模型自动过滤出'仅单模型'分组"
- "需要复核的 needs_review issue 高亮提示"

### 5. 结果标准化增强

当前系统已经汇总 `issues`，后续可扩展问题字段：

| 字段 | 说明 |
|---|---|
| status | 三级判定 Pass / Fail / Review |
| category | 格式、完整性、标准符合性、逻辑一致性等 |
| confidence | high / medium / low / needs_review |
| rule_code | 命中的规则编号 |
| evidence | 判定依据 |
| acceptance_status | 是否接受，供人工处理 |

这些字段可以继续存入 `ai_result` JSONB，不影响现有任务模型。

## 六、新增其他文档审查场景的方法

当系统需要新增其他文档审查场景时，按以下顺序扩展：

1. 定义场景名称和审查边界，例如“合同审查”“需求规格说明书审查”“测试报告审查”。
2. 创建该场景对应的规则库，按“通用规则、专项规则、输出规范”拆分。
3. 上传规则文件，规则中写清适用条件、检查项、输出要求。
4. 创建审查场景并关联规则库。
5. 使用现有 `/api/v1/reviews/execute` 提交文档、场景和模型。
6. 如该文档类型不能用一级标题切分，再扩展新的 `parser_strategy` 或 `chunk_strategy`。
7. 如该场景存在大量专项规则，再扩展规则元数据和规则调度层。

这样可以保证新增场景优先通过配置完成，只有共性能力不足时才改后端。

| 能力 | 当前状态 | 建议处理 |

## 八、结论

试验大纲智能审查作为当前 AI 文件审查系统中的一个"可配置审查场景"落地。系统已经具备：
- 规则库 / 审查场景 / 按章节调度规则；
- 收敛性 AI 调用（统一参数 + 结构化输出）；
- 逐章节审查（每章节 = 一次调用，命中规则 + 本章节切片 + 引用章节切片）；
- 跨切片 fingerprint 去重与枚举归一化；
- 限流(429)抗性（尊重 `Retry-After` + 单章节失败隔离）；
- 失败可重审、Excel 导出、WebSocket 实时进度。

试验大纲的业务差异沉淀在规则库的元数据（`rule_code` / `rule_type` / `sections` / `keywords`）和规则正文中；当前 Markdown 规则文件可按 `##` 拆成多条规则，上传后每条上传业务规则都会在 CHUNK 检查矩阵中至少形成一条判定。内置 `R-Q` 文字质量/图表编号检查只在发现问题或需复核时显示。主流程通用、对新场景零侵入。

后续重点：DO-160G 检查单 Excel 转规则工具、章节映射层、多模型对比前端 UI。
